    - HebbianReinforcer: increases concept.strength on reuse
    - SelfPlayEngine: autonomous knowledge discovery
      Generates hypotheses from existing concepts
      Proves or disproves them
      Stores new proven knowledge automatically

class RIRLoop:

  def process(self, raw_input: str) -> HSCIResponse:
    """
    Complete end-to-end processing of any input.
    Steps:
      1. Perceive: build PerceptionMap from raw_input
      2. Recall: query KnowledgeBase for relevant concepts
      3. Reason: build candidate solution using concepts
      4. Verify: prove candidate with Z3
         - If VALID: proceed to step 5
         - If INVALID: self-correction loop (max 5 attempts)
         - If still INVALID after 5 attempts: return 
           honest "cannot prove" response with partial trace
      5. Learn: extract concept, update weights, store episode
      6. Return: HSCIResponse with answer + full proof trace
    """

HSCIResponse must contain:
  - answer: the proven solution
  - proof_trace: step-by-step reasoning
  - concepts_used: which concepts were applied
  - verification_status: PROVEN / FAILED / PARTIAL
  - confidence: float 0.0-1.0
  - new_concept_learned: bool
  - correction_attempts: int

    Use regex + simple NLP patterns. No external NLP libraries.
    Build your own pattern matching for:
      - "X is/= N" → entity extraction
      - "find/calculate/solve X" → target extraction
      - "X% of Y" → percentage relation
      - "X + Y" / "sum of" → addition relation
      - "X * Y" / "product of" → multiplication relation
      - "X - Y" / "difference" → subtraction relation
    """

class Z3Verifier:

  def verify(self, spec: FormalSpec) -> VerificationResult:
    """
    Core verification method.

    FormalSpec contains:
      - constraints: list of Z3 constraint strings
      - variables: dict of {name: type} for Z3 declarations
      - target: the variable being solved for
      - expected: expected value if known

    Process:
      1. Declare Z3 variables from spec.variables
      2. Add all constraints to Z3 solver
      3. Call solver.check()
      4. If SAT:
         - Extract model values
         - Verify target matches expected (if provided)
         - Extract proof trace
         - Return VerificationResult(valid=True, ...)
      5. If UNSAT:
         - Extract minimal unsatisfiable core
         - Identify which constraint caused failure
         - Return VerificationResult(valid=False, 
             counterexample=..., failure_reason=...)

    CRITICAL: Never catch Z3 exceptions silently.
    Always surface the failure reason for learning.
    """

  def build_proof_trace(self, solver, model) -> ProofTrace:
    """
    Extract structured proof trace from Z3 model.
    ProofTrace contains:
      - steps: ordered list of reasoning steps
      - contributing_constraints: which constraints mattered
      - variable_assignments: final values of all variables
      - structural_pattern: abstract pattern for concept extraction
        e.g. "multiplicative_reduction_3_variables"
    """

class ProofGuidedUpdater:

  def update_on_success(self, 
                         perception: PerceptionMap,
                         proof_trace: ProofTrace):
    """
    When Z3 proves a solution valid, update perception weights
    based on WHAT IN THE PERCEPTION LED TO THE CORRECT FORMALIZATION.

    Algorithm:
    1. Extract contributing features from proof_trace
       Features = entity types + relationship types + axiom choice
       that appeared in both perception AND proof
    
    2. For each contributing feature f:
       weights[f] += learning_rate * proof_trace.confidence
       # strengthen: this perception feature led to valid proof
    
    3. For each non-contributing feature f in perception:
       weights[f] -= learning_rate * 0.1
       # slightly weaken: this was noise in the perception
    
    4. Normalize all weights to range [0.0, 1.0]
    
    5. Persist updated weights to synaptic_weights.json

    learning_rate = 0.15 (configurable in config.py)
    """

  def update_on_failure(self,
                         perception: PerceptionMap, 
                         counterexample: Counterexample):
    """
    When Z3 returns counterexample, update weights to
    avoid this wrong perception in the future.

    Algorithm:
    1. Extract features that led to the WRONG formalization
    
    2. For each wrong feature f:
       weights[f] -= learning_rate * 0.3
       # stronger penalty for wrong formalization
    
    3. Extract what the counterexample IMPLIES is correct
       e.g. counterexample shows variable should be Real not Int
       → strengthen Real-type perception features
    
    4. Persist updated weights
    """

class ConceptExtractor:

  def extract(self, 
              proof_trace: ProofTrace,
              examples_so_far: list) -> ConceptDefinition | None:
    """
    Given a proof trace, extract an abstract reusable concept.
    Uses Inductive Logic Programming principles.

    Algorithm:
    1. Get structural_pattern from proof_trace
       e.g. "result = component_a - (component_a * rate)"
    
    2. Check if this pattern already exists in concept library
       - If exact match: strengthen existing concept (Hebbian)
       - If partial match: create specialization concept
       - If no match: create new concept
    
    3. For new concept creation:
       a. Generalize the specific values to variables
          "5000 - (5000 * 0.20)" → "base - (base * rate)"
       b. Find the minimal rule covering all seen examples
          (use version space: most-specific to most-general)
       c. Generate Z3 template from generalized rule
       d. Infer domain from entity types
       e. Find generalizes_to by checking ontology for
          structurally similar existing concepts
    
    4. Return ConceptDefinition or None if not novel enough
    
    IMPORTANT: Only create new concept if:
      - Pattern seen at least 2 times (avoid overfitting)
      - Generalized rule is simpler than specific examples
      - Z3 can verify the generalized rule
    """

class SelfPlayEngine:

  def run_discovery_cycle(self, iterations: int = 100):
    """
    Autonomous knowledge discovery loop.
    Runs in background thread when system is idle.

    For each iteration:
    1. GENERATE hypothesis:
       - Pick 2 random concepts from library
       - Compose them into a new hypothesis
       - e.g. PERCENTAGE + ADDITION → 
         "total = base + (base * rate)"
    
    2. SOLVE the hypothesis:
       - Use reasoning engine to find satisfying values
    
    3. VERIFY with Z3:
       - If VALID: this is a new provable relationship
       - If INVALID: this combination is impossible
         (store as IMPOSSIBILITY — also valuable knowledge)
    
    4. LEARN:
       - Valid: store as new concept with strength=0.3
         (lower initial strength since not from real problem)
       - Invalid: store counterexample in episode memory
    
    5. LOG discovery to console:
       "[SELF-PLAY] Discovered: PERCENTAGE_ADDITION concept"
       "[SELF-PLAY] Disproved: division-by-zero impossibility"
    """

  def generate_targeted_practice(self, weak_concept: str):
    """
    Generate practice problems for concepts with low strength.
    Like a human studying weak areas.
    
    - Find concept with strength < 0.4
    - Generate 5 problems that require this concept
    - Solve and learn from each
    - Concept strength should increase after practice
    """

SECTION 7: IMPLEMENTATION PRINCIPLES — NEVER VIOLATE THESE
═══════════════════════════════════════════════════════════════════

PRINCIPLE 1: PROOF OR SILENCE
  Never return an answer that Z3 has not verified.
  If Z3 cannot prove it, return "cannot prove" with partial trace.
  A system that admits uncertainty is more intelligent
  than one that guesses confidently.

PRINCIPLE 2: PROOF-GUIDED LEARNING ONLY
  Neural weights update ONLY from Z3 proof traces.
  Never update weights randomly.
  Never use cross-entropy loss on token prediction.
  Every weight change must be traceable to a specific proof.

PRINCIPLE 3: CONCEPTS NOT PARAMETERS
  Intelligence lives in the concept library, not weight counts.
  A small concept library that is CORRECT is better than
  a large weight matrix that is APPROXIMATE.

PRINCIPLE 4: FULL TRANSPARENCY
  Every answer must include full reasoning trace.
  No black box outputs.
  A human must be able to follow every step.

PRINCIPLE 5: GRACEFUL DEGRADATION
  When system cannot solve a problem:
    - Return what it CAN prove (partial solution)
    - Explain what it is missing
    - Store the failure as a learning opportunity
    - Generate a self-play hypothesis to fill the gap
  Never crash. Never hallucinate. Never guess.

PRINCIPLE 6: SEPARATION OF CONCERNS
  Neural layer: ONLY perception. Never reasoning.
  Symbolic layer: ONLY verification. Never perception.
  Learning layer: ONLY weight updates. Never solving.
  These boundaries must never be crossed.
